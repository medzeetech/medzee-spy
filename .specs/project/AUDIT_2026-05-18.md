## Causa-raiz: uazapi paid (`naorpedroza`)

Os 3 endpoints que precisamos estão TODOS instáveis nesse tier:

| Endpoint | Estado observado |
|---|---|
| `POST /instance/create` | ✅ 200 OK consistente |
| `POST /instance/connect` (QR) | ✅ 200 OK consistente |
| `POST /webhook` | ✅ 200 OK (após fix do body em 2026-05-18) |
| `GET /instance/status` | ✅ 200 quando token ativo, 401 quando rotacionado |
| `POST /chat/find` (recém-conectado) | ❌ 500 nos primeiros minutos pós-connect |
| `POST /message/find` (chats antigos) | ✅ 200 quando testado isolado via curl |

**Hipótese**: o tier do `naorpedroza` parece ser um shared cluster onde `/webhook` está globalmente broken. `/chat/find` precisa de "warm-up" de history sync que pode demorar 5-15min em vez dos 60s que esperamos.

## O que dá pra fazer hoje (sem mudar provider)

**A.** Aguardar mais tempo entre connect e extract.
Mudar `_POST_CONNECTED_DELAY_S` de 60s pra 180s. UX ruim (3min de espera).

**B.** Estado de "primeira coleta agendada".
Pós-connect, mostrar pro user "Coletando histórico — primeiro relatório disponível em ~3 min". Polling.

**C.** Botão manual "Tentar coletar agora" no Dashboard.
Em vez de auto-fire, deixa o user clicar quando quiser. Cada clique tenta `/chat/find` (com retry 105s).

**D.** Aceitar relatório parcial quando 0 dados.
Mostrar empty state honesto com sugestão ("aguarde mais conversas", "verifique conexão", "tente novamente em N min") em vez de relatório com números alucinados.

## O que daria salto qualitativo (mudança maior)

**E.** Trocar de provider de WhatsApp.
- **Evolution API** (self-hosted, sem quota) — não tem 500 random
- **WhatsApp Cloud API oficial** (Meta) — mais estável, mas exige aprovação Meta
- **Outro tier uazapi** (`uazapi.com` próprio em vez de `naorpedroza`) — sem garantia

**F.** Aceitar que demo "5min" morreu e re-vender o produto.
- Posicionar como "dashboard contínuo": user conecta, deixa coletando 7 dias, relatório fica pronto. Atrasa o aha-moment mas resolve o problema técnico.

## Última coisa honesta

O backend tem bons fundamentos. Auth funciona, RLS protege, pipeline de relatório está abstraído (F4 reusou F3 inteiro). **O bloqueio é exclusivo da uazapi paid** — não é arquitetura, não é frontend, não é LLM.

Se a uazapi paid funcionasse como anunciado, o sistema renderia o relatório real em < 5min. Mas como ela não funciona, todo investimento em otimizar warmup/retry/fallback é gambiarra em volta de um provider inadequado.
